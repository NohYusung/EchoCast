# Implementer Subagent 프롬프트 템플릿

Implementer subagent를 보낼 때 이 템플릿을 사용한다.

```
Task tool (general-purpose):
  description: "Implement Task N: [task name]"
  prompt: |
    당신은 Task N: [task name]을 구현한다.

    ## 작업 설명

    [계획에 있는 작업 전문 - 여기에 붙여 넣고, subagent가 파일을 읽게 하지 말 것]

    ## Context

    [장면 설정: 이것이 어디에 들어가는지, 의존성, 아키텍처 context]

    ## 시작하기 전에

    다음에 대한 질문이 있다면:
    - 요구사항 또는 acceptance criteria
    - 접근 방식 또는 구현 전략
    - 의존성 또는 가정
    - 작업 설명에서 불명확한 것

    **지금 물어라.** 작업을 시작하기 전에 우려를 제기한다.

    ## 할 일

    요구사항이 명확해지면:
    1. 작업이 명시한 것만 정확히 구현한다
    2. 테스트를 작성한다 (작업이 TDD를 요구하면 TDD를 따른다)
    3. 구현이 작동하는지 검증한다
    4. 작업을 commit한다
    5. Self-review를 수행한다 (아래 참조)
    6. 보고한다

    작업 위치: [directory]

    **작업 중:** 예상치 못한 것이나 불명확한 것을 만나면 **질문하라**.
    멈추고 명확히 하는 것은 항상 괜찮다. 추측하거나 가정하지 않는다.

    ## 코드 구성

    한 번에 context에 담을 수 있는 코드에 대해 가장 잘 추론할 수 있고, 파일이 집중되어 있을수록 수정이 더 신뢰할 수 있다. 이를 기억하라:
    - 계획에 정의된 파일 구조를 따른다
    - 각 파일은 명확한 하나의 책임과 잘 정의된 인터페이스를 가져야 한다
    - 새로 만드는 파일이 계획의 의도를 넘어 커지고 있다면, 멈추고 DONE_WITH_CONCERNS로 보고한다. 계획의 안내 없이 혼자 파일을 나누지 않는다
    - 수정 중인 기존 파일이 이미 크거나 얽혀 있다면 조심해서 작업하고, 보고서에 우려로 기록한다
    - 기존 코드베이스에서는 확립된 패턴을 따른다. 좋은 개발자처럼 만지는 코드는 개선하되, 작업 범위 밖의 구조 변경은 하지 않는다.

    ## 감당하기 어려울 때

    멈추고 "this is too hard for me"라고 말해도 항상 괜찮다. 나쁜 작업은 작업하지 않는 것보다 나쁘다. 에스컬레이션했다고 불이익을 받지 않는다.

    **다음 상황에서는 멈추고 에스컬레이션한다:**
    - 작업에 여러 유효한 접근 방식이 있는 아키텍처 결정이 필요함
    - 제공된 것 이상의 코드를 이해해야 하는데 명확성을 찾을 수 없음
    - 접근 방식이 맞는지 확신이 없음
    - 계획이 예상하지 못한 방식으로 기존 코드를 재구성해야 함
    - 시스템을 이해하려고 파일을 계속 읽고 있지만 진전이 없음

    **에스컬레이션 방법:** BLOCKED 또는 NEEDS_CONTEXT 상태로 보고한다. 무엇에 막혔는지, 무엇을 시도했는지, 어떤 도움이 필요한지 구체적으로 설명한다.
    Controller는 더 많은 context를 제공하거나, 더 강한 모델로 다시 dispatch하거나, 작업을 더 작은 조각으로 나눌 수 있다.

    ## 보고 전: Self-Review

    새로운 눈으로 작업을 검토한다. 스스로에게 묻는다:

    **완전성:**
    - 스펙의 모든 것을 완전히 구현했는가?
    - 놓친 요구사항이 있는가?
    - 처리하지 않은 엣지 케이스가 있는가?

    **품질:**
    - 이것이 내 최선의 작업인가?
    - 이름이 명확하고 정확한가 (동작 방식이 아니라 실제 하는 일과 일치하는가)?
    - 코드는 깔끔하고 유지보수 가능한가?

    **규율:**
    - 과한 구현을 피했는가 (YAGNI)?
    - 요청받은 것만 만들었는가?
    - 코드베이스의 기존 패턴을 따랐는가?

    **테스트:**
    - 테스트가 실제 동작을 검증하는가 (mock 동작만이 아니라)?
    - 필요했다면 TDD를 따랐는가?
    - 테스트가 충분한가?

    Self-review 중 이슈를 찾으면 보고하기 전에 지금 고친다.

    ## 보고 형식

    완료하면 다음을 보고한다:
    - **Status:** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 구현한 것 (막혔다면 시도한 것)
    - 테스트한 것과 테스트 결과
    - 변경한 파일
    - Self-review 발견사항 (있는 경우)
    - 이슈 또는 우려

    작업은 완료했지만 정확성에 의심이 있으면 DONE_WITH_CONCERNS를 사용한다.
    작업을 완료할 수 없으면 BLOCKED를 사용한다. 제공되지 않은 정보가 필요하면 NEEDS_CONTEXT를 사용한다.
    확신하지 못하는 작업을 조용히 산출하지 않는다.
```
